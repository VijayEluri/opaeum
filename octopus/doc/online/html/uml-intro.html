The Unified Modeling Language

The following text is translated and adapted from 'Praktisch UML' by Jos Warmer and Anneke Kleppe, published by Addison-Wesley in the Netherlands.

The visual language Unified Modeling Language (UML) is a collaborative product of a large group of companies. UML is a standard that was developed by leading technologists in the object oriented world, as an answer to a `request for proposals' of the Object Management Group (OMG).

The origin of UML

At the start of the nineties the first object oriented systems development methods were being published. Among those the Object Modeling Technique (OMT), as developed by Jim Rumbaugh et al., the method developed by Grady Booch, and Ivar Jacobson's OOSE method have been very popular. They decided to merge their methods, and submitted the result as a proposal to the request for proposals that the OMG issued at the time.

Together with five other submitters they worked on a final submission that was accepted as the standard by the OMG in September 1997. One of the additions to the original UML proposal was the Object Constraint Language (OCL), which was part of the IBM/ObjecTime proposal.

A language, not a method

UML is not a method, it does not prescribe how people should work. The only thing that is standardized by UML are the work products (diagrams, etc.) that should be produced.

The diagrams

UML offers a wide variety of diagrams. Together they can be used to develop a model of a software system.

  • The use-case diagram shows how the system can be used by external actors, like e.g. human beings.
  • The class diagram shows the static structure of the software system using classes and their relationships.
  • The object diagram shows the static structure of the software system using objects (or instances) and their relationships.
  • The sequence diagram shows the order in time of messages send between the various objects in the system.
  • The collaboration diagram shows how a group of object collaborate to accomplish a goal.
  • The state diagram shows the states which one object can be found in during its lifetime.
  • The activity diagram shows activities that are being performed by (part of) the system, including potential parallelism.
  • The component diagram shows the division of the complete system in separate components, including the relationships between the components.
  • The deployment diagram shows how software components are deployed in a certain hardware configuration.

Apart from the diagrams UML offers a textual language that can be used to specify conditions and rules in the diagrams. This language is called the Object Constraint Language (OCL).
