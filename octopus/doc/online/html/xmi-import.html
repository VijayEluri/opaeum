XMI Import

  • Version : 1.2
  • Author : Mick Baggen

Use

To start the import use the XMI import button on the tool bar or the 'Import XMI file' entry in the Octopus menu. Select the XMI file in the file selection dialog and press 'Open'.

Description

The XMI Import module is based on a set of XSL stylesheets for conversion of an UML XMI file to a set of Octopus UML files. It utilizes the eXtensible Stylesheet Language Transformation (XSLT) technology. The XSLT specification can be found at www.w3.org. This set of stylesheets was previously part of the UML2Octopus distribution. UML2Octopus was a stand-alone tool for transforming Poseidon XMI files into the Octopus format.

The current version of the XMI Import module is tested with the XMI output generated by the following UML design tools:

  • Poseidon for UML Standard Edition 2.4.1 (URL: http://www.gentleware.com)
  • Rational Rose Enterprise Edition 2002.05.20 (URL: http://www.ibm.com/software/rational)
  • MagicDraw 7.5 beta (URL: http://www.magicdraw.com)

  • Objecteering
  • Enterpise Architect

Limitations and workarounds

  • Octopus only allows packages as top level UML modelelement, no other modelelements are allowed (such as classes). When creating a model in Poseidon, the user should therefore create one or more child packages in the top level Model, and add all model elements to these child packages. Of course, these packages can also contain other packages, forming a hierarchy.

  • Octopus does not support 'package' visibility. the XMI import module surpresses output for a visibility of this type. The user should not employ this visibility type in Poseidon.

  • Octopus does not support sequences of multiplicity ranges, such as [1..3,4..6]. The user should not create such ranges in his UML design tool.

  • Octopus only allows names for UML modelelements (including names of Packages, Classes etc) that contain upper and lowercase characters, numbers and an underscore. The name may not start with a number, and empty names or names containing containing whitespace, a dot etc. are not allowed. the XMI import module generates warning that indicate these errors to the user before actually encountering them in Octopus. The user should only use allowed names in his UML design tool.

  • Since Octopus is an OCL tool, it only has the following basic predefined primitive types: Integer, Real, Boolean, String and OclVoid. However, Octopus allows the user to provide a mapping (using the typeconversions.properties file) from the type system used in his favorite UML design tool to its OCL-based type-system. This properties file is provided by the XMI import module, and uses a default mapping adapted for Poseidon for UML (which is a Java-based system):

The user is free to edit the properties file. The XMI import module will automatically include the changed file with the generated .uml files in the 'model' directory.

XMI limitations and workarounds

  • Poseidon does not provide support for automatic generation of package and element imports. However, Poseidon allows the user to add a Dependency from a package to another package or class. the XMI import module converts this construct to an explicit package or element import statement in the Octopus .uml file. It is not necessary (and actually an error) to declare an import dependency to the default java.lang package that is automatically created by Poseidon. The third sample xmi file DemoImport contains a demonstration of the package and element import.

  • Poseidon does not provide support for (native) Enumerations. However, Poseidon allows the definition of a Datatype with a stereotype <<enumeration>>. the XMI import module converts this construct as if it were a native Enumeration.

  • Poseidon allows the user to add constraints to some modelelements using a text-based contraints editor. The user should only add constraints to a Class, Datatype, Interface or AssociationClass, NOT to an attribute or operation of these classifiers. The user should also set the language of each of these contraints to 'OCL' (the choice of Sterotype is not relevant). Constraints without this language attribute, or added to other model elements, will not be processed by the XMI import module.

Known Problems and Omissions

The current version of the XMI import module is a beta version. The following issues are known. If you encounter any other problems, please inform the author of the XMI import module by sending an email to octopus@klasse.nl.

  • Due to the directory tree generation mechanism of the XMI import module, each package or classifier in the XMI file is required to have a unique name. Not providing an unique name will cause the transformation to malfunction.

  • The XMI import module requires the UML namespace-value in the XMI file to be defined as "org.omg.xmi.namespace.UML", the value generated by Poseidon. If you want to use the XMI export of another UML modeling tool, you should manually edit the XMI file and adjust the UML namespace value to the one above. If this is not done, the transformation process will generate empty output.

  • In case of multiple inheritance from a common superclass higher up in the inheritance tree, the XMI import module generates multiple identical <specializes> statements. The user should eliminate the redundant classnames manually.
