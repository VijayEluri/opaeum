The Textual Representation of UML

Octopus uses its own textual representation of UML. Note that this notation is NOT part of the UML standard. In the Octopus representation each package is written in a separate file, which should have the file extension `.uml'. All UML files must be placed under the dedicated model folder in the Octopus project, otherwise Octopus will not recognize them. Subfolders are considered to indicate subpackages. A subfolder should contain a file with the `.uml' extension, and the package that is specified in this file must have the same name as the folder.

The UML Editor

The UML Editor is a simple editor for UML files. It offers syntax highlighting and a rudimentary form of code completion. You can open the UML editor on any file with extension '.uml'.

Rules for an UML file

The rules are explained here by example. For those that prefer a structure definition, the BNF syntax definition is available.
  1. A UML file must begin with "<package> DIRNAME", where DIRNAME is the name of the folder in which the file is placed.
  2. A UML file must end with "<endpackage>".
  3. A UML file may contain a number of
    • classifier entries, and/or
    • association entries.
  4. A classifier entry is either
    • a class entry,
    • an interface entry,
    • a datatype entry, or
    • an enumeration type entry.
  5. An association entry is either
    • a simple association entry, or
    • an association class entry.
  6. A simple association entry consists of two ends separated by '-', '<-', '->', or '<->'. From version 1.1.0 the arrowheads are used to indicate navigability of the association end. An end must define
    • the visibility (+, -, or #),
    • the class,
    • the rolename (if no rolename is present, use <noName>), and
    • the multiplicity ([0..1],[1], [0..*], [1..*],or [3..12]).

      An example, where both ends are navigable, is:

+ Chapter.prevChap[0..1] <-> + Chapter.nextChap[0..1];

An example, where only the preveChap end is navigable, is:

+ Chapter.prevChap[0..1] <- + Chapter.nextChap[0..1];
  1. An association class entry combines the elements in a simple association entry and a class entry.
Further details can be found in the example.

Example Model File

<package> Book
<class> Bookpart
 <attributes>
+ title: String;
 + nrOfPages: Integer;
 - /startPageNr: Integer;
 <operations>
+ determineStartPageNumber(): Integer;
 <endclass>

<class> Prependix <specializes> Bookpart
 <attributes>
+ kind: PrependixKind;
 <operations>
+ needsEmptyLastPage(): Boolean;
 + isFirstBookpart(): Boolean;
 <endclass>

<class> Chapter <specializes> Bookpart
 <attributes>
+ author: String;
 <operations>
+ determineDurationTillFinished(from: Date): Period;
 <states>
ready;
 inwriting;
 inwriting::draft;
 inwriting::finaldraft;
 <endclass>

<class> Appendix <specializes> Bookpart <implements> TwoColumnPart
 <attributes>
+ kind: AppendixKind;
 - twoColumn: Boolean;
 <operations>
+ needsEmptyLastPage(): Boolean;
 <endclass>

<interface> TwoColumnPart
 <operations>
+ transformToTwoColumn();
 + transformToOneColumn();
 <endinterface>

<datatype> Period
 <attributes>
+ nrOfDays: Integer;
 + nrOfWorkingDays: Integer;
 <enddatatype>

<datatype> Date
 <attributes>
+ day: Integer;
 + month: Integer;
 + year: Integer;
 <enddatatype>

<enumeration> PrependixKind
 <values>
contents;
 figures;
 tables;
 preface;
 <endenumeration>

<enumeration> AppendixKind
 <values>
index;
 bibliography;
 glossary;
 <endenumeration>

<associationclass> ChapterDependency
+ Chapter.sourceChapter[0..1] <-> + Chapter.dependantChapters[0..*]
 <attributes>
sameAuthor: Boolean;
 sameSubject: Boolean;
 <endassociationclass>

<associations>
+ Chapter.prevChap[0..1] <-> + Chapter.nextChap[0..1];
 + Chapter.prevChap[0..1] <-> + Appendix.nextApp[0..1];
 + Appendix.prevApp[0..1] <-> + Appendix.nextApp[0..1];
 + Prependix.prevPrep[0..1] <-> + Chapter.nextChap[0..1];
 + Prependix.prevPrep[0..1] <-> + Prependix.nextPrep[0..1];
 <endpackage>
