Octopus Project Properties

Projects with an Octopus nature have a number of properties. These can be accessed through the Properties option in the context menu in the Eclipse Navigator.

The Octopus project properties are the names of the two special folders within the project: the model folder and the expression folder. You can select the directories that contain the appropriate files. If you want to rename the current model or expression folder, enter the new name in the text field and check the 'Rename existing folder?' box.

Code Generation Properties

There is a larger number of properties that can be set to customize the Java code generation.

The first properties you may set are the folders where the generated code should be placed. You can indicate the folder where the code directly linked to the model should be, and you can indicate the folder where extra utilities, like code for the OCL standard library, should be put.

The code generation can be customized to a larger extend by turning features on or off. The following options are avaiable:

  • Generate interfaces and 'internal' package: if selected each package in the model will be transformed into a package containing interfaces and a subpackage which contains the actual classes that implement the interfaces.
  • Generate code for OCL expressions: if selected all correct OCL expressions will be translated into code.

    A precondition will be transformed into an assert statement. (Be sure to set the Compiler compliance level to 1.4. This option can be found in Windows>Preferences>Java>Compiler>Compliance and classfiles.) Note that postconditions are not transformed into code. A body expression will, of course, be transformed into the body of the operation.

    An invariant will be transformed into a method that throws an InvariantException when the invariant is broken. This method can be called from any place you feel is appropriate. There will also be a method available called checkAllInvariants that may be called whenever appropriate. This method calls in its turn all methods that check invariants. Its result is a list of InvariantError objects. Each InvariantError holds a reference to the instance that has broken the invariant and a message that tells you what went wrong.

    A derivation rule is transformed into a getter method for the attribute or association end for which the rule was defined. Note that there is no setter method. An init expression will, of course, be transformed into an initialisation of the field that corresponds to the attribute or association end.

  • Generate separate classes for postgeneration editing: if selected each package in the model will be transformed into a package holding empty classes for each model element in the package as well as a subpackage that holds the classes that contain the generated code. The empty classes can be used to add fields and methods that were not fit to be in the model. These classes will not be overwritten when you generate code for a second time.

    Note that is option may be used together with the option to generate interfaces. In that case a package P that contains classes A and B will be transformed into a Java package P that contains interfaces IA and IB and a subpackage called internal. The internal package contains the empty placeholders for extra methods and fields called A and B and a subpackage called generated. The generated package contains the classes that hold the generated code. They are called AGEN and BGEN.

  • Generate model visitor and accept operations: if selected a visitor interface will be created (and placed in the utilities package) that is able to visit all classes in the model. This interface is called using the project name: I<projectname>Visitor. In every class an accept operation will be added that ensures that every link in the class will also be visited. Note that some objects may therefore be visited more than once.
  • Generate user interface prototype: if selected classes will be generated that implement a prototype of a user interface for the system. The possibility to create objects of a certain class in this prototype will be available for every class or datatype that is selected in the second tab of the properties page (called User Interface). See Using the Generated User Interface for an explanation on how to use the prototype.
  • Generate XML storage facility: if selected a simple storage facility will be generated. The objects created in the system can be stored on file using an XML format. The XML format is based on the UML metamodel and is therefore equal for all models.
