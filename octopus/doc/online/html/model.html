The UML model

OCL expressions always need a set of model elements like classes, attributes, operations, etc. This set is called the UML model or the UML part of the model. Usually the UML model is created in a UML modeling tool, of which a large number is available on the market, open source or otherwise.

Octopus users may create their UML model in two ways:

  • The model can be created within Octopus itself using a textual representation of UML,
  • The model can be exported from the UML modeling tool into an XMI format and then imported in the Octopus project.

    To start the import use the XMI import button on the tool bar or the 'Import XMI file' entry in the Octopus menu. (Note that the buttons in the tool bar do not appear when Octopus is loaded the first time. Use Window>Customize Perspective>Other to make them appear.) The import module in Octopus was created by Mick Baggen using XSLT.

Please, be aware that currently each modeling tool uses its own dialect of the UML interchange format XMI. It is therefore very difficult to create an XMI import functionality that works correctly for each UML model. If you encounter problems, please send us the model in XMI format and we will try to adjust the import module.

Mapping Basic Types

The basic types available in some UML modeling tools are sometimes different from the OCL basic types (String, Integer, Real, and Boolean). Octopus uses the OCL basic types only. Other types need to be included in the UML model.
 Existing basic types in a UML modeling tool can be mapped onto the OCL basic types. Create a file named 'typeconversions.properties' in the model folder. The entries in this file need to be in the format 'external-type=ocl-type'. Note that spaces before or after the equal sign will be regarded part of the type name. A typeconversions.properties file that maps Java basic types to OCL basic types will typically contain the following entries:

int=Integer

boolean=Boolean

float=Real

void=OclVoid
